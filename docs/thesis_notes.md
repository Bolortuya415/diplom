## 1. Sensitive Content Classifier (Custom Trained Model)

### Purpose
Мэдрэмтгий агуулга ангилагч нь хэрэглэгчийн оруулсан текстийг аюулгүй эсэхийг шалгаж, үзэн ядалт, дарамт, ялгаварлан гадуурхалт, өөрийгөө гэмтээх шинж илэрвэл тохирох хариу үйлдэл үзүүлдэг.

### Why It Is Needed
Хүйсийн тэгш байдал, нийгмийн хүртээмжтэй байдлын чатбот нь мэдрэмтгий сэдвүүдтэй харьцдаг учир хэрэглэгчийн аюулгүй байдлыг хангах аюулгүй байдлын давхарга зайлшгүй шаардлагатай. Ялангуяа өөрийгөө гэмтээх шинж тэмдэг илэрсэн тохиолдолд яаралтай тусламжийн мэдээлэл өгөх нь чухал.

### How It Works
1. Хэрэглэгчийн текстийг цэвэрлэж, Монгол Кирилл тэмдэгтийг хэвийн болгоно
2. TF-IDF (Term Frequency — Inverse Document Frequency) аргаар тэмдэгтийн n-грамм шинж чанар гаргана
3. Логистик регресс (Logistic Regression) загвараар 5 ангилалд хуваана: safe, hate_speech, harassment, discrimination, self_harm
4. Загвар нь магадлалын оноо гаргаж, итгэлцлийн босго хэрэгжүүлнэ

### Why This Method Was Chosen
- **TF-IDF + Logistic Regression**: Энэ нь текст ангилалт хийхэд зориулагдсан сонгодог, хялбар, тайлбарлахад хялбар арга юм
- Олон хэлний BERT загвартай харьцуулахад илүү хурдан сурч, оюутны төвшинд хэрэгжүүлэхэд тохиромжтой
- Тэмдэгтийн n-грамм (char_wb) нь Монгол хэлний морфологид илүү тохиромжтой — үгийн нөхцөл, залгаварыг илүү сайн илрүүлдэг
- GridSearchCV-ээр гиперпараметрийн оновчлол хийж, 5-fold cross-validation ашиглан загварыг баталгаажуулсан

### Limitations
- Сургалтын өгөгдөл нь 100 жишээтэй, харьцангуй бага хэмжээтэй
- Мэргэжлийн annotation баг ашиглаагүй — оюутны бүтээсэн шошго
- Контекстоос хамааралтай тохиолдолд (жишээ нь: дарамтын талаар асуух vs дарамтлах) алдаа гарч болно
- Шинэ төрлийн хортой агуулгыг таних чадвар хязгаарлагдмал

## 2. RAG Pipeline (Retrieval-Augmented Generation)

### Purpose
RAG pipeline нь хэрэглэгчийн асуултад итгэмжлэгдсэн эх сурвалжаас хамааралтай мэдээллийг олж, LLM-ээр хариулт үүсгэх бүрэн процесс юм.

### Why It Is Needed
LLM-үүд нь сургалтын өгөгдөлд байхгүй мэдээлэл "зохиож" (hallucination) болох сул талтай. RAG нь энэ асуудлыг баримт бичгийн эх сурвалжаар шийдэж, хариултыг найдвартай эх сурвалжид суурилуулдаг. Ингэснээр хэрэглэгч эх сурвалжийг шалгах боломжтой болно.

### How It Works
1. **Баримт бичиг оруулах**: PDF/TXT файлаас текст гаргаж авна (PyMuPDF)
2. **Хэсэглэх (Chunking)**: Текстийг 500 тэмдэгтийн хэсгүүдэд хуваана (50 тэмдэгтийн давхцалтай)
3. **Embedding**: sentence-transformers (paraphrase-multilingual-MiniLM-L12-v2) загвараар вектор болгоно
4. **Индекслэх**: FAISS (Facebook AI Similarity Search) санд хадгална
5. **Хайлт**: Хэрэглэгчийн асуултыг embed хийж, cosine similarity-аар хамгийн ойр 5 хэсгийг олно
6. **Хариулт үүсгэх**: Олдсон хэсгүүдийг LLM-д контекст болгон өгч, эх сурвалжтай хариулт үүсгэнэ

### Why This Method Was Chosen
- **paraphrase-multilingual-MiniLM-L12-v2**: 50+ хэл дэмжих, 384 хэмжээст embedding, зөөврийн компьютер дээр ажиллах хэмжээтэй
- **FAISS**: Meta AI-ийн open-source вектор хайлтын сан, хурд, өргөтгөх чадвараараа давуу
- **Chunk size 500**: Контекстийн бүрэн байдал ба хайлтын нарийвчлалын тэнцвэр
- **Top-k=5**: LLM-ийн token лимитэд багтаж, хангалттай контекст өгөх

### Limitations
- Монгол хэлний embedding чанар англи хэлнээс доогуур байж болно
- Хэсэглэлтийн стратеги нь өгүүлбэрийн бүтцийг тасалж болно
- Зурагт мэдээлэл (хүснэгт, диаграмм) боловсруулахгүй

## 3. Answer Generation (LLM Integration)

### Purpose
RAG pipeline-аас авсан контекстийг ашиглан хэрэглэгчийн асуултад эх сурвалж бүхий хариулт үүсгэнэ.

### Why It Is Needed
Вектор хайлт нь зөвхөн хамааралтай текстийн хэсгүүдийг олж өгдөг, гэхдээ хэрэглэгчид хялбар, уншихад тохиромжтой хариулт хэрэгтэй. LLM нь олдсон мэдээллийг нэгтгэж, ойлгомжтой хариулт үүсгэдэг.

### How It Works
- System prompt нь Монгол хэлээр зааварчилгаа агуулна
- Олдсон эх сурвалжуудыг [1], [2] гэх мэт дугаарлан prompt-д оруулна
- LLM (GPT-4o-mini) хариулт үүсгэж, эх сурвалжийн дугаарыг inline-аар оруулна
- Temperature=0.2 (бага) ашигласан — бодитой хариултыг чухалчилна

### Why This Method Was Chosen
- **GPT-4o-mini**: Хямд, хурдан, multilingual чадвар сайтай
- **Low temperature**: Бүтээлч бус, бодитой хариулт үүсгэхэд тохиромжтой
- **Inline citations**: Хэрэглэгч хариултын аль хэсэг ямар эх сурвалжаас авсныг шууд харах боломжтой

### Limitations
- Гадаад API-аас хамааралтай (интернэт холболт шаардлагатай)
- API зардал гардаг (хэдий ч GPT-4o-mini хямд)
- LLM нь заримдаа эх сурвалжид байхгүй мэдээлэл нэмж болно

## 4. FastAPI Backend

### Purpose
Бүх компонентуудыг нэгтгэн REST API болгож, frontend-тэй холбох гүүр.

### Why It Is Needed
Frontend ба ML компонентуудын хооронд стандартчилсан интерфейс хэрэгтэй. FastAPI нь хурдан, автомат баримтжуулалттай (OpenAPI), Pydantic баталгаажуулалттай.

### How It Works
- `/api/chat` — Хэрэглэгчийн асуулт хүлээн авч, classifier → RAG → хариулт process
- `/api/ingest` — Шинэ баримт бичиг оруулах
- `/api/feedback` — Хэрэглэгчийн үнэлгээ хадгалах
- `/api/health` — Системийн статус шалгах
- SQLite database-д бүх log хадгалагдана

### Why This Method Was Chosen
- **FastAPI**: Python-ийн хамгийн хурдан web framework, async дэмжих
- **SQLite**: Нэмэлт сервер суулгах шаардлагагүй, оюутны төвшинд тохиромжтой
- **Service pattern**: Код модульчлагдсан, тест хийхэд хялбар

## 5. React Frontend

### Purpose
Хэрэглэгчтэй харилцах вэб интерфейс — чат, эх сурвалж, аюулгүй байдлын анхааруулга.

### Why It Is Needed
Чатбот нь техникийн бус хэрэглэгчдэд зориулагдсан учир ойлгомжтой, хялбар интерфейс чухал. Эх сурвалжийн панел нь хариултын найдвартай байдлыг баталгаажуулахад тусалдаг.

### How It Works
- React компонент бүтэц: ChatPage, AdminPage, MessageBubble, SourcePanel, SafetyWarning
- Vite ашиглан хурдан хөгжүүлэлт
- API proxy тохиргоогоор backend-тэй холбогдоно

## 6. Evaluation Framework

### Purpose
Системийн гурван түвшний үнэлгээ: retrieval чанар, хариултын чанар, аюулгүй байдал.

### Metrics Used

**Retrieval Quality:**
- Precision@k: Олдсон хэсгүүдийн хэд нь хамааралтай вэ?
- Hit@k: Хамгийн багадаа нэг хамааралтай хэсэг олдсон уу?
- MRR (Mean Reciprocal Rank): Эхний хамааралтай хэсэг хэддүгээрт байрласан бэ?

**Answer Quality (LLM-as-Judge):**
- Relevance: Хариулт асуултыг хариулсан уу?
- Faithfulness: Хариулт эх сурвалжид суурилсан уу?
- Clarity: Хариулт тодорхой, ойлгомжтой юу?
- Safety: Хариулт хүндэтгэлтэй, аюулгүй юу?

**User Evaluation:**
- 10 асуулт бүхий судалгааны маягт (Likert 1-5 шкала)
- Сэтгэл ханамж, хэрэглэхэд хялбар байдал, итгэлцлийн гурван бүлэг

### Why This Approach
- Олон түвшний үнэлгээ нь системийн аль хэсэг сайн/муу ажиллаж байгааг тодорхойлдог
- LLM-as-judge нь хүний үнэлгээтэй корреляцитай гэж судалгаагаар батлагдсан
- Хэрэглэгчийн судалгаа нь бодит хэрэглээний үнэлгээ өгдөг

## Thesis Defense Tips

### "Яагаад бүтэн LLM сургаагүй вэ?" гэсэн асуултад:
> "LLM сургах нь тэрбум параметр, олон сая өгөгдөл, олон долоо хоногийн GPU хугацаа шаарддаг.
> Оронд нь бакалаврын ажлын хүрээнд бодитой, практик ач холбогдолтой мэдрэмтгий агуулга
> ангилагч загварыг сургаж, аюулгүй байдлын давхарга болгон ашигласан. Энэ нь бодит салбарын
> практикт нийцсэн — аж ахуйн нэгжүүд ч мөн адил бэлэн LLM ашиглаж, дагалдах загваруудыг
> тусгайлан сургадаг."

### "100 жишээ хангалттай юу?" гэсэн асуултад:
> "Энэ нь proof-of-concept түвшний хэрэгжүүлэлт юм. Бодит production системд хэдэн мянган
> жишээ хэрэгтэй. Гэхдээ TF-IDF + Logistic Regression нь бага өгөгдөлтэй сайн ажилладаг
> гэж олон судалгаагаар баталгаажсан. Цаашид өгөгдлийн хэмжээг нэмэгдүүлэх, мэргэжлийн
> annotator ашиглах зэргээр сайжруулах боломжтой."

### "RAG яагаад ашигласан бэ?" гэсэн асуултад:
> "LLM-үүд нь сургалтын өгөгдөлд байхгүй мэдээллийг 'зохиох' (hallucination) сул талтай.
> Монгол хэлний, тухайлбал хууль, бодлогын мэдээлэл LLM-ийн сургалтын өгөгдөлд бага байдаг
> учир RAG нь итгэмжлэгдсэн эх сурвалжаас мэдээлэл авч, хариултыг найдвартай болгодог."
