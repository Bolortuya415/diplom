# Boloroo Системийн Диаграмуудын Нэгдсэн Тайлбар

**Зориулалт:** Дипломын тайланд оруулах академик хэв маягийн нэгдмэл тайлбар. Энэхүү баримт бичиг нь `docs/diagrams/` хавтсан доторх 7 диаграмыг нэгтгэн, тус бүрийн ач холбогдол, харилцан хамаарал, дипломын ажлын бүлэг тус бүрд хэрхэн ашиглах талаарх зөвлөмжтэй танилцуулна.

---

## Удиртгал

Boloroo (UI дээр «Тэгшбот» хэмээн нэрлэгдэх) нь хүйсийн тэгш эрх, ялгаварлан гадуурхалт, хөгжлийн бэрхшээлтэй иргэдийн эрхийн талаар Монгол хэл дээр мэдээлэл, зөвлөгөө өгдөг RAG-д суурилсан туслах чатбот юм. Систем нь дараах нийтлэг архитектурын зарчмуудад тулгуурласан:

- **Layered architecture** — UI давхарга, API давхарга, business logic давхарга, persistence давхарга нь тус тусдаа.
- **Service-oriented decomposition** — `chat_service.py`, `ingest_service.py` зэрэг тусгайлан хариуцлага хүлээдэг бүрэлдэхүүнтэй.
- **Local-first deployment** — Cloud үйлчилгээ ашиглахгүй, бүх боловсруулалт хэрэглэгчийн машин дээр явагдана.
- **Retrieval-Augmented Generation** — гадаад мэдлэгийн санг ашиглан LLM-ийн hallucination бууруулах.

Энэхүү бүлэг доторх 7 диаграм нь системийн архитектур, өгөгдлийн урсгал, хэрэглэгчийн харилцан үйлдэл, деплой бүтэц, мэдлэгийн санг шинэчлэх процессыг хамтад нь зурдаг.

---

## I. Зураг 1 — Системийн Ерөнхий Архитектур

[Линк: `docs/diagrams/01_system_architecture_mn.md`]

Уг диаграм нь Boloroo системийн **гурван давхар архитектурыг** зурдаг:

1. **Frontend давхарга (React + Vite)** — LandingPage, ChatPage, AdminPage гэсэн гурван үндсэн хуудастай.
2. **Backend давхарга (FastAPI + Python 3.14)** — REST API эндпойнтоор бүх дотоод үйлчилгээг нэгтгэдэг.
3. **Persistence ба гадаад үйлчилгээ** — FAISS вектор индекс, SQLite база, ML загваруудын pickle файл, дотроо ажиллах Ollama LLM (`qwen2.5:7b`) гэсэн дөрвөн төрлийн нөөцтэй.

Системийн гол давуу тал бол **decoupled service pattern** юм. RAG retrieval нь LLM-ээс тусдаа модуль (`rag/pipeline.py`), generation нь RAG-аас тусдаа (`rag/generator.py`). Үүний үр дүнд бичих туршилт хийх, нэг бүрэлдэхүүнийг солих, өргөтгөх боломж нь маш сайн.

**Дипломын ажилд:** 3-р бүлэг «Системийн дизайн ба архитектур»-т оруулах зөвлөмжтэй. Энэ диаграмаар «бид яагаад FastAPI-г сонгосон, яагаад FAISS-ыг сонгосон» гэсэн технологийн сонголтын үндэслэлийг тайлбарлах боломжтой.

## II. Зураг 2 — RAG Дамжуулалтын Урсгал

[Линк: `docs/diagrams/02_rag_pipeline_mn.md`]

RAG нь системийн **гол шинжлэх ухааны үндэс** юм. Уг диаграм нь хоёр үе шатыг ялгаж харуулна:

- **Indexing үе шат (offline)** — баримт бичиг → текст гаргалт → cleaning → FAQ эсвэл char-level chunking → embedding → FAISS-д хадгалах.
- **Query үе шат (online)** — хэрэглэгчийн асуулт → category prefix → embedding → FAISS search → re-ranking (FAQ boost +0.12) → top-2 chunk → FAQ shortcut эсвэл LLM-аар хариулт үүсгэх.

Ялангуяа **FAQ-ийн фонголтыг зөвхөн асуултын текстээр embedding хийдэг** (хариулт нь metadata-д үлддэг) гэдэг нь стандарт RAG-аас давсан энд хэрэгжсэн **онцлог инноваци**. Энэ нь хэрэглэгчийн query (асуулт хэлбэртэй) болон FAQ entry хоорондын cosine similarity-ыг ихэсгэдэг.

**Тоон үнэлгээ:** Live тестээр «Хүйсийн тэгш байдлын тухай хуулиар юу хамгаалагддаг вэ?» гэсэн query-д top-1 score 0.949-д хүрсэн нь FAQ-aware chunking ба embedding стратеги ажиллаж байгаагийн нотолгоо.

**Дипломын ажилд:** 4-р бүлэг «RAG-ийн хэрэгжүүлэлт ба онцлог»-т оруулах. Энэ нь дипломын ажлын **техникийн гол хувь нэмэр** болж тэмдэглэгдэх ёстой.

---

## III. Зураг 3 — Хэрэглэгчийн Харилцан Үйлдлийн Дараалал (Sequence)

[Линк: `docs/diagrams/03_sequence_chat_flow_mn.md`]

Sequence diagram нь нэг чат хүсэлтийн дотор **6-аас 8 алхамын дараалал**-ыг зурдаг:

1. Identity check → шууд танилцуулга буцаах
2. Capability check → шууд чадварын танилцуулга
3. Greeting check → шууд мэндчилгээ
4. Safety classifier → 5 ангиллыг шалгах
5. Crisis indicator-аар false positive-ыг бууруулах downgrade логик
6. Vague query check → тодруулах асуулт буцаах
7. RAG retrieval → top-2 chunk
8. FAQ fast-path эсвэл LLM-аар хариулт үүсгэх

Энэ дараалал нь **«гэдсэн шилэн optimizer pattern»** хэв маягтай. Хамгийн хямд (regex match) шалгуурууд эхэнд, хамгийн үнэтэй (LLM call) хамгийн төгсгөлд. Иймэрхүү design нь CPU laptop-д **дундаж хариу хугацааг 5–10 секундээс 0.05–2 секунд хүрэхэд** боломжтой нөхцөл бүрдүүлдэг (FAQ хариулт бол шууд буцдаг).

**Дипломын ажилд:** 5-р бүлэг «Гүйцэтгэлийн оптимизаци ба system design»-т оруулах. Энэ диаграмыг ашиглан *«яагаад routing priority дараалал ингэж сонгогдсон бэ?»* гэсэн зөвтгөх асуултанд хариу болно.

## IV. Зураг 4 — Use Case Диаграм

[Линк: `docs/diagrams/04_use_case_mn.md`]

Use case диаграм нь системийн **functional requirement**-уудыг хэрэглэгчийн талаас зурдаг:

- **Энгийн хэрэглэгч (5 use case):** ангилал сонгох, асуулт асуух, хариулт авах, эх сурвалж харах, санал өгөх.
- **Админ хэрэглэгч (5 use case):** баримт бичиг нэмэх, жагсаалт харах, устгах, эрүүл мэнд шалгах, статистик харах.
- **Систем (5 internal use case):** classifier-аар шалгах, vector хайлт, LLM хариу үүсгэх, лог хадгалах, FAISS index шинэчлэх.

Use case-ийн «include» хамаарлаар *«асуулт асуух»* нь *«classifier шалгах»* + *«vector хайлт»* + *«LLM хариу»* гурвыг шаардаж буйг харуулдаг.

**Дипломын ажилд:** 2-р бүлэг «Шаардлагын анализ»-т оруулах. Энэ нь системийн зорилго, хэрэгцээ, бизнесийн logic-ийг **техникийн шалгуураас тусгаар** тайлбарлахад хэрэглэгдэнэ.

## V. Зураг 5 — Өгөгдлийн Урсгал (Data Flow Diagram)

[Линк: `docs/diagrams/05_data_flow_mn.md`]

Data flow diagram нь **нэг асуултын мөчлөгийн дотор өгөгдөл хэрхэн төлөвөө өөрчилж буйг** дагалздаг:

- Текст оролт → JSON request → Pydantic validated object → cleaned text
- Cleaned text → TF-IDF vector + safety result (зэрэгцээ)
- Cleaned text → multilingual MiniLM embedding (384-dim float32)
- Embedding → FAISS query → top-2 chunks dict
- Chunks → numbered context string + source citation list
- Context + system prompt → Ollama HTTP body → raw LLM answer
- Raw answer → cleaned answer (citation header strip) → JSON response

Бас **хадгалагдах өгөгдөл** болох `chat_logs` хүснэгт болон `feedback` хүснэгтийн харилцан хамаарлыг харуулна.

**Дипломын ажилд:** 4-р бүлгийн «Өгөгдлийн боловсруулалт» дэд хэсэгт орох. Энэ диаграм нь **анонимтай өгөгдөл хадгалалт** (хэрэглэгчийн ID байхгүй) гэсэн privacy-friendly шинж чанарыг бас тусгана.

## VI. Зураг 6 — Деплой Диаграм

[Линк: `docs/diagrams/06_deployment_mn.md`]

Deployment diagram нь **физик орчныг** зурдаг:

- **Хэрэглэгчийн ноут буукт** 3 параллел процесс ажиллана: Vite (5173), Uvicorn (8000), Ollama (11434).
- **Файлын persistence**: FAISS index (5 MB), chunks pickle (3 MB), SQLite (122 KB), ML моделүүд (~110 KB), эх баримт (5 PDF + 7 TXT, нийт ~3 MB).
- **RAM шаардлага**: ~5–6 GB (Ollama 4.7 GB + sentence-transformer 470 MB + бусад).
- **Сонголт болох Docker compose** — frontend container + backend container, гэхдээ Ollama-руу хүрэхэд `host.docker.internal` тохируулах шаардлагатай.

**Дипломын ажилд:** 6-р бүлэг «Системийн деплой ба тохиргоо»-нд оруулах. Хамгаалалтын өдөр **«Ollama сервер заавал ажиллаж байх ёстой»** гэсэн critical operational requirement-ыг тэмдэглэх.

## VII. Зураг 7 — Админ Баримт Бичиг Оруулах Урсгал

[Линк: `docs/diagrams/07_admin_ingest_flow_mn.md`]

Энэ flowchart нь **мэдлэгийн санг динамикаар шинэчлэх процессыг** харуулна:

1. Админ AdminPage-ээр PDF/TXT файл сонгож upload.
2. Backend file extension validate.
3. `data/raw/`-д хадгалах.
4. `IngestService` → `RAGPipeline.ingest_document()`.
5. PyMuPDF/text loader → cleaning → FAQ detection → chunking (FAQ-aware эсвэл character-based).
6. Embedding → FAISS-д add → файл руу хадгалах.
7. SQLite-д `documents` болон `chunks` хүснэгтэд метадата бичих.
8. `IngestResponse` JSON буцаах.

**Гол ач холбогдол:** систем нь **зөвхөн анхны seed data-д хязгаарлагдахгүй**. Шинэ хууль батлагдсан үед админ хууль бичгийг upload хийхэд систем шууд тэр хуулиар асуултанд хариулж эхэлдэг.

**Дипломын ажилд:** 5-р бүлэг «Knowledge base management» дэд хэсэгт оруулах. Бас «future improvement»-д **transactional ingestion** (FAISS + SQL атомик) санал нэмэх хэрэгтэй.

## Диаграмуудын Харилцан Хамаарал

Долоон диаграм нь дараах байдлаар нэг бүтэц үүсгэдэг:

| Диаграм | Гол асуулт | Дипломын бүлэг |
|---------|-----------|----------------|
| 1 — System architecture | Систем юунаас тогтох вэ? | Дизайн |
| 2 — RAG pipeline | RAG яаж ажилладаг вэ? | Хэрэгжүүлэлт |
| 3 — Sequence flow | Нэг асуулт яаж боловсруулагддаг вэ? | Хэрэгжүүлэлт |
| 4 — Use case | Хэн юу хийдэг вэ? | Шаардлагын анализ |
| 5 — Data flow | Өгөгдөл яаж хувирдаг вэ? | Хэрэгжүүлэлт |
| 6 — Deployment | Систем хаана яаж ажилладаг вэ? | Деплой |
| 7 — Admin ingest | Мэдлэг яаж шинэчлэгддэг вэ? | Хэрэгжүүлэлт |

**Дипломын тайланд** ашиглах зөвлөмж:
- Бүх диаграмыг **тус бүр өөрийн бүлэгт** оруулахаас илүү, бүлгийн логикийн дагуу **2–3 диаграмыг хамт байрлуулах** нь уншигчид илүү ойлгомжтой.
- Жишээ: Хэрэгжүүлэлт бүлэгт Зураг 2 + 3 + 5 хамтдаа байрлуулж, *«технологийн уг шинж — өгөгдөл — дараалал»* гэсэн гурван өнцгөөс RAG-ийг тайлбарлах боломжтой.

## Тайлбарын Стандарт

Бүх диаграм нь дараах **3-хэсгийн тайлбар** хэлбэртэй:

1. **Mermaid диаграм** — markdown render-ийн боломжтой кодын блок.
2. **Дэлгэрэнгүй тайлбар** — техникийн нарийвчлан тайлбар, файл/мөрийн ишлэлтэй.
3. **Дипломын тайланд ашиглах тайлбар** — академик хэв маягийн тайлбар, бүлэг/хэсэг тус бүрд тохируулсан зөвлөмж.
4. **Хамгаалалтын үеэр тайлбарлах богино хувилбар** — 3–5 өгүүлбэрт хураангуй, амьд хариулахад бэлэн.

Энэхүү бүтэц нь:
- Markdown-аар render хийгдвэл диаграм харагдах.
- PDF-руу export хийхэд ердийн академик хэв маягтай.
- Хамгаалалтын үед оратор бэлэн «scriptтай» болох.

## Дүгнэлт

Boloroo системийн архитектурын чанар нь **олон давхрын тайлбарт хэвлэгдэх боломжтой** байх — энэ нь зөвхөн ажилладаг код биш, **зураглагдсан, баримтжсан, хамгаалалтад бэлэн системийн** шинж чанар. Долоон диаграм нь дипломын ажилд:

1. Системийн **зорилго ба функциональ хэрэгцээг** тайлбарлах (Use case).
2. Системийн **дотоод бүтэц ба технологийг** үндэслэх (System architecture, Deployment).
3. **Гол шинжлэх ухааны үндэс** болох RAG-ийг үзүүлэх (RAG pipeline, Data flow).
4. Системийн **Гүйцэтгэлийн оптимизацийг** тайлбарлах (Sequence flow).
5. **Удаан хугацааны хэрэглээг** дэмжих knowledge management процессыг тайлбарлах (Admin ingest).

Эдгээр диаграм нь хамтдаа нэг **complete architectural narrative** үүсгэн дипломын комисст системийн чанарыг бодиттойгоор үнэлэх үндэслэлийг бий болгоно.
